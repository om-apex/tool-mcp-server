## Problem Statement

The MCP server generates all task IDs as `TASK-nnn` regardless of task type. With the TASK-504 naming convention restructure, enhancement/feature tasks should use `DEV-nnn` and issue tasks should use `ISSUE-nnn` prefixes. The current `get_next_task_id()` function in `supabase_client.py` only queries `TASK-%` IDs and increments a single counter.

Additionally, the handoff system uses a singleton `session_handoff` table (always `id=1`) that gets upserted on every save, with the old value archived to `session_handoff_history`. This design is being replaced by project-keyed local `handoff.md` files, making `get_session_handoff` unnecessary and `save_session_handoff` reducible to a simple history-append.

## Context & Background

- TASK-504 (complete) restructured naming conventions in rules and skills
- Task IDs in Supabase remain `TASK-nnn` — this task changes the ID generation
- `get_next_task_id()` in `supabase_client.py:314-347` uses client-side max-ID-plus-one logic
- Handoff singleton pattern in `supabase_client.py:785-869` uses `session_handoff` table with CHECK constraint `id=1`
- Local `handoff.md` files per project are the new primary handoff mechanism (per TASK-504 design session)
- `session_handoff_history` has no `project_code` column — all history entries are undifferentiated

## Proposed Solution

### 1. Task ID Generation

Replace `get_next_task_id()` with `get_next_task_id(task_type: str)`:

| task_type | Prefix | Example |
|-----------|--------|---------|
| `enhancement` | `DEV-` | `DEV-506` |
| `feature-request` | `DEV-` | `DEV-507` |
| `issue` | `ISSUE-` | `ISSUE-001` |
| `dev` | `DEV-` | `DEV-508` |
| `manual` | `DEV-` | `DEV-509` |

**Sequence numbering:** Use a single shared counter across all prefixes. Query ALL task IDs (`TASK-%`, `DEV-%`, `ISSUE-%`), extract the max numeric suffix, and increment. This prevents collisions (e.g., DEV-506 and ISSUE-506) and maintains a monotonic global order. The counter continues from wherever TASK-nnn left off.

**Backward compatibility:** All existing `TASK-nnn` IDs remain valid. The `get_pending_tasks` tool must accept `TASK-nnn`, `DEV-nnn`, and `ISSUE-nnn` in the `task_id` filter. The `update_task`, `complete_task`, and `update_task_status` tools already accept any string ID — no changes needed there.

### 2. Handoff Schema Changes

- **DROP** the `session_handoff` table (singleton `id=1` pattern no longer needed)
- **ADD** `project_code TEXT` column to `session_handoff_history`
- **Simplify** `save_session_handoff`:
  - Remove the "read existing → archive to history → upsert singleton" flow
  - Replace with a single INSERT into `session_handoff_history`
  - Add required `project_code` parameter
- **Remove** `get_session_handoff` tool entirely (local handoff.md is primary)
- **Update** `get_handoff_history` to require `project_code` parameter
