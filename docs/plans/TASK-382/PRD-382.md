# PRD — TASK-382: Harden get_pending_tasks MCP Tool

**Task ID:** TASK-382
**Type:** Enhancement
**Source:** Nishad
**Date Created:** 2026-03-08
**Status:** prd-to-review

## Problem Statement

The `get_pending_tasks` MCP tool returns unbounded result sets when called
without filters. With 380+ tasks in the system, an unfiltered call returns
350K+ characters of JSON — exceeding the MCP tool's token limit and causing
tool failures. This is a recurring problem: any caller that forgets to add
a filter triggers it.

The root cause is that `get_pending_tasks` has no required parameters, no
result limit, and returns full task records (`select("*")`). By contrast,
`get_task_queue` already has a `limit` parameter (default 10) and returns
truncated records — but `get_pending_tasks` has neither safeguard.

Additionally, there is no text search capability — callers cannot search
across task descriptions and notes without downloading everything.

## Context & Background

- `get_pending_tasks` calls `sb_get_tasks()` in `supabase_client.py:159-198`
  which does `client.table("tasks").select("*")` with optional filters
- `get_task_queue` already has limit (default 10, fetches 3x, sorts, truncates)
- `complete_task` and `force_complete` also call `sb_get_tasks()` with no filters
  to find a single task by ID — wasteful but low risk (internal only, result
  not returned to caller)
- `get_decisions_history` has the same pattern — `select("*")` with optional
  filters and no limit — but decisions table is smaller (~200 records)
- Several AI Quorum tools (`list_quorum_sessions`, `get_quorum_logs`,
  `get_quorum_model_performance`) also have no hard limits, but those tables
  can grow large over time

## Proposed Solution

### REQ-01: Harden `get_pending_tasks` tool
- Add `limit` parameter (default 10, max 50) to both tool schema and `sb_get_tasks()`
- Add `search` parameter for full-text search across `description` and `notes`
  (uses Supabase `or(description.ilike.%term%, notes.ilike.%term%)`)
- Add `task_id` parameter for fetching a single task by ID (exact match)
- Enforce that at least one filter or `task_id` is provided — reject calls
  with zero filters by returning an error message (not an exception)
- Default to excluding `complete` status when no status filter is provided
  (matches `get_task_queue` behavior)
- Keep `select("*")` — full records are needed for task detail views;
  the limit cap prevents unbounded output

### REQ-02: Fix internal unbounded `sb_get_tasks()` calls
- `complete_task` (line 242) and `force_complete` (line 353) call
  `sb_get_tasks()` with no filters just to find one task by ID.
  Replace these with a direct `get_task_by_id()` helper that does
  `client.table("tasks").select("*").eq("id", task_id).limit(1).execute()`
- This is an internal optimization — doesn't change the tool interface

### REQ-03: Add limit to `get_decisions_history`
- Add `limit` parameter (default 10, max 50) to `get_decisions_history`
  tool schema and `get_decisions()` in `supabase_client.py`
- Decisions table is smaller but growing — add limit now to prevent
  the same problem later

### REQ-04: Add limits to AI Quorum diagnostic tools
- Add or enforce limits on `list_quorum_sessions` (already has `limit` param,
  verify it's applied at DB level), `get_quorum_logs` (already has `limit`),
  `get_quorum_model_performance` (no limit)
- Cap any missing limits at reasonable defaults

### REQ-05: Update skill files and rules
- Audit all skill files and rules that call `get_pending_tasks`
- Update calls to include appropriate filters (typically `status` filter)
- Document the new `search`, `task_id`, and `limit` parameters in relevant
  skill files where `get_pending_tasks` is referenced

## Acceptance Criteria

- [ ] `get_pending_tasks` with no arguments returns an error message requiring at least one filter
- [ ] `get_pending_tasks` with a filter returns at most 25 results by default
- [ ] `get_pending_tasks(limit=10)` works; `limit=50` is capped to 100
- [ ] `get_pending_tasks(search="quorum")` returns tasks with "quorum" in description or notes
- [ ] `get_pending_tasks(task_id="TASK-382")` returns exactly that task
- [ ] `complete_task` and `force_complete` no longer call unbounded `sb_get_tasks()`
- [ ] `get_decisions_history` with no filters returns at most 50 results
- [ ] AI Quorum diagnostic tools all have limit caps applied
- [ ] No skill file calls `get_pending_tasks` without at least one filter argument
- [ ] MCP server starts and all tools register without errors

## Out of Scope

- Changing the Supabase schema or adding database indexes
- Pagination (cursor-based) — limit cap is sufficient for current scale
- Changing `get_task_queue` — it already has proper limits
- Changing the Owner Portal UI's task queries

## Open Questions

1. Should `get_pending_tasks` default to excluding `complete` tasks (like `get_task_queue` does), or should it require the caller to specify a status filter? Defaulting to non-complete seems more useful and prevents the largest result sets.
Answer: Yes, exclude complete tasks by default and I would suggest we use in-progress and pending as the default status filter.
2. Is 25 the right default limit, or should it be higher (e.g., 50)? The main constraint is MCP tool output token limits.
Answer: I would go with 10 default limit and 50 max limit.
3. New request from Nishad: add similar filters and defaults to other mcp tools that return large unbounded result sets.
