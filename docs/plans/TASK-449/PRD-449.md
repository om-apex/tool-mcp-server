# PRD — TASK-449: Make project_code Mandatory in add_task MCP Tool

## Problem Statement

Newly created tasks in the Om Apex task system consistently have blank `project_id` values because the `add_task` MCP tool does not expose a `project_code` parameter. This makes it impossible to filter, group, or report on tasks by project without manually updating each record after creation. Every task belongs to a project, and this association should be captured at creation time — not retroactively.

## Context & Background

The infrastructure for project-task association already exists:

- **`projects` table** (Owner Portal Supabase `hympgocuivzxzxllgmcy`): Contains all 15 projects with a `project_folder` TEXT column that maps 1:1 to `$PROJECT_CODE` values used in Claude Code sessions (e.g., `mcp-server`, `portal`, `ai-quorum`).
- **`tasks.project_id`** (UUID FK to `projects.id`): The column exists in the tasks table but is almost never populated because the MCP tool doesn't accept it.
- **`$PROJECT_CODE` environment variable**: Set by the `claude()` shell function in `~/.zshrc` at session start. Always available in Claude Code sessions.

The `add_task` tool currently accepts: `description`, `category`, `company`, `priority`, `notes`, `task_type`, `source`, `prd_path`, `commit_refs`, `issue_ref`. Neither `project_code` nor `project_id` is among them.

The `update_task` tool similarly lacks a `project_code` parameter, so even correcting the omission after creation requires direct Supabase API calls.

## Source Data

Nishad's request (verbatim):
> I am looking at some newly created tasks and I see that the project code is left blank in most of them. Why don't we make this project code a mandatory field to be populated with the right project code. So, confirm the project code with Nishad before creating the task. If we are able to derive project code from a valid source or using an algorithm that will make my life a bit easier. ## Proposed Solution

### 1. MCP Tool Changes (`add_task`)

Add a `project_code` parameter to the `add_task` tool:
- **Required field** — tool call fails if omitted
- Accepts human-friendly project codes (e.g., `mcp-server`, `portal`, `ai-quorum`)
- Validates against `projects` table: looks up `project_folder` to find the UUID
- Special mapping: `root` -> `project_folder = 'om-apex'`
- Stores the resolved UUID in `tasks.project_id`
- Returns the project name in the success response for confirmation

### 2. MCP Tool Changes (`update_task`)

Add a `project_code` parameter to the `update_task` tool:
- **Optional field** — only needed when correcting or changing project association
- Same validation and UUID lookup as `add_task`
- Allows fixing existing tasks that have null `project_id`

### 3. Claude Code Workflow Rules

Update rules to define the project_code confirmation protocol:
- Use `$PROJECT_CODE` as the default suggestion
- Before calling `add_task`, Claude confirms the project_code with Nishad if the task context suggests a different project than the current `$PROJECT_CODE`
- If the project is clearly derivable from context (task explicitly references a known project), Claude proposes it directly
- The confirmation can be a simple inline check: "Creating this as an `mcp-server` task — correct?"

### 4. Documentation Fix

The `DB-SCHEMA-QUICK-REF.md` incorrectly documents the column as `project_code TEXT` when the actual column is `project_id UUID` (FK to `projects.id`). This must be corrected.

## Acceptance Criteria

- [ ] `add_task` requires `project_code` parameter; calls without it return a validation error
- [ ] `add_task` resolves `project_code` to a UUID via `projects.project_folder` lookup and stores it in `tasks.project_id`
- [ ] `add_task` returns an error with a list of valid project codes if an invalid code is provided
- [ ] `update_task` accepts an optional `project_code` parameter with the same lookup behavior
- [ ] `root` project code correctly maps to `project_folder = 'om-apex'` in the projects table
- [ ] Claude Code workflow rules require project_code confirmation before task creation
- [ ] `DB-SCHEMA-QUICK-REF.md` correctly documents `project_id UUID` (not `project_code TEXT`)
- [ ] MCP server deploys successfully to Render after changes
- [ ] Existing tasks with null `project_id` are unaffected (no backfill in this task)

## Out of Scope

- Backfilling `project_id` on existing tasks — separate task if desired
- Adding `project_code` to `get_pending_tasks` filter options — separate enhancement
- Displaying project code in the Owner Portal task table UI — separate enhancement
- Changes to the `projects` table schema — no modifications needed
- Adding new project codes to the `projects` table — existing entries cover all current projects

## Open Questions

1. Should the `project_code` values be hardcoded in the tool's enum validation, or should the tool query the `projects` table dynamically at call time? Dynamic is more maintainable but adds a Supabase call per task creation.
2. For the `root` -> `om-apex` mapping: should this be handled as a special case in the tool code, or should a `root` alias be added to the projects table?
3. Should we backfill existing tasks with null `project_id` as a follow-up task?
